OnCommand Performance Manager banner displays 'Data collection is taking too long on Cluster'

212 Ratings Log in to rate this article KB ID: 1012696 Version: 6.0 Published date: 09/03/2015 Views: 2905

Environment

OnCommand Performance Manager (OPM) 1.x and 2.x

Symptoms

OnCommand Performance Manager presents a banner stating Data collection is taking too long on Cluster or a banner stating Cluster ( ) is unreachable. Performance Manager is no longer monitoring this cluster.

Example message from the ocfserver-debug logs:

2015-04-24 07:05:56,195 DEBUG [system] [reconcile-0] [c.n.ocf.collector.OcieJmsListener] OCIE JMS notification message received: {Error0_ClusterManagementIP= , ErrorCount=1, Error0_Port=443, DatasourceID=4, DatasourceName= , UpdateTime=1429883940563 , Success=false, PollStartTime=1429883940563, DurationTime=415591, Error0_Type=NETWORK_ACCESS_FAILURE, Error0_Message= Communication problem with the cluster : , Failed to download Archive files . Error: ' : Download wait timed out after: 240000 milliseconds.' on try 37 out of 37, NotificationType=PACKAGE_COMPLETED, PackageName=netappstorageperformance, SampleTime=1429883940563, MessageType=PACKAGE_NOTIFICATION} 2015-04-24 07:05:56,198 ERROR [system] [org.springframework.jms.listener.DefaultMessageListenerContainer#0-1] [com.netapp.ipc.jms.OCIE_Events] ***************************************************************************** Collection duration is greater than 5 minutes o - Cluster: [ ] o - Duration: 415591 (ms) *****************************************************************************

Cause

Causes include, but are not limited to the following:

  1. The quantity of nodes and/or clusters surpassed the support configurations.
  2. The cluster being inaccessible to Performance Manager due to connectivity issues or user privilege issues.
  3. The SPI service in the cluster is disabled or in a bad state.

Solution

Solution to Cause 1:

Confirm that the instance of OPM is following the 'Supported cluster configurations' in the Release Notes.

For more information, see OnCommand Performance Manager for Clustered Data ONTAP.

Solution to Cause 2:

Make sure the username through which the cluster was authenticated in OPM has ontapi, ssh, and http privileges to the cluster.

This can be done through the OnCommand System Manager (OCSM) or the CLI. For example:

::> security login create -username opm -authmethod password -role admin -application http ::> security login create -username opm -authmethod password -role admin -application ontapi ::> security login create -username opm -authmethod password -role admin -application ssh ::> security login show -vserver -username opm Vserver: Authentication Acct UserName Application Method Role Name Locked ---------------- ----------- -------------- ---------------- ------ opm http password admin no opm ontapi password admin no opm ssh password admin no 3 entries were displayed. Check also if a firewall exist in the environment it should allow traffic on http

This will need to be performed on the cluster specifically:

::*> vserver services web access show -vserver -name spi Vserver Type Service Name Role -------------- -------- ---------------- ---------------- admin spi none

Assign the admin role to be able to access the spi data.

::*> vserver services web access create -vserver -name spi -role admin

Check the following:

::*> vserver services web access show -vserver -name spi Vserver Type Service Name Role -------------- -------- ---------------- ---------------- admin spi admin

Solution to Cause 3:

Run the the following commands to restart the SPI service:

::> vserver services web modify spi -vserver -enabled false ::> vserver services web modify spi -vserver -enabled true
